# pp-collision-diffusion — Claude Code context

This repository trains a **parton-conditioned diffusion model** for full-event generation at the LHC. Architecture: PET_pp_parton — Point Edge Transformer with per-parton cross-attention. Trained on matched (MG5 parton-level, Pythia hadron-level) event pairs to learn P(final state | hard scatter).

**Two training jobs are the immediate goal: E022 and E023.** Both train on the same W' → 4q signal grid dataset. Follow the steps below in order.

---

## Step 0 — Discover the cluster environment

Before doing anything else, run these to understand what you're working with:

```bash
sinfo                          # available partitions and node types
sacctmgr show qos              # available QOS names (if any)
sacctmgr show account          # your account(s)
nvidia-smi                     # GPU type and count per node
which conda || which mamba     # check conda availability
```

Record the GPU partition name, your account name, and how many GPUs are available per node. You will need these when filling in the `FILL_IN` placeholders in `submit/*_portable.sh`.

---

## Step 1 — Install the Python environment

Run once:

```bash
bash setup/install_env.sh
```

This creates a conda env called `pp-diffusion` with:
- Python 3.10
- TensorFlow 2.15.0
- numpy, h5py
- pythia8 (for data generation)
- Horovod (attempted — requires MPI; falls back to single-GPU if unavailable)

**If Horovod is not installed**, training runs on a single GPU automatically. The training scripts detect this via a try/except wrapper and substitute a single-GPU stub. No code changes needed.

**If MPI is available** and you want multi-GPU training, load your cluster's MPI module before running `install_env.sh`, e.g.:
```bash
module load openmpi   # cluster-specific
bash setup/install_env.sh
```
Then update `--ntasks` and `--gpus` in the submit scripts accordingly and replace the `python3 -u` launch line with `horovodrun -np NGPU python3 -u`.

---

## Step 2 — Generate the W' signal data

The training data is a 12×12 mass grid of W'→XY→4q signal events, generated by `data_generation/generate_wprime_signal.py` using Pythia8. Each mass point produces `signal_mX{mX:04d}_mY{mY:04d}.hdf5` (100k events, ~500 MB each).

**Fill in the `FILL_IN` placeholders** in `data_generation/submit_wprime_grid_portable.sh`, then submit the array job:

```bash
mkdir -p logs
sbatch data_generation/submit_wprime_grid_portable.sh
```

This submits 144 array tasks (one per mass point). Each takes ~5-10 min on a CPU node. Monitor with `squeue -u $USER`. When all tasks complete, check the output directory contains 144 `.hdf5` files:

```bash
ls FILL_IN_DATA_DIR/signal_mX*.hdf5 | wc -l   # should be 144
```

The checkpoint directories contain:
- `pet_pp.weights.h5` — model weights (the critical file)
- `training_state.json` — epoch counter and val_loss history
- `normalisation_stats_*.json` — data normalization stats (E023 only; E022 recomputes from data)

If NERSC access is unavailable, training will start from epoch 0. Delete any partial checkpoint dirs so the training scripts don't pick up a corrupt state:
```bash
rm -rf FILL_IN_DATA_DIR/checkpoints_bsm_grid/bsm_grid_event_c_layers4
rm -rf FILL_IN_DATA_DIR/checkpoints_bsm_grid/bsm_grid_event_c_stage1
```

## Step 4 — Configure and submit training jobs

### Fill in the portable submit scripts

Edit both `submit/submit_e022_portable.sh` and `submit/submit_e023_portable.sh`. Replace every `FILL_IN_*` placeholder:

| Placeholder | What to put |
|-------------|-------------|
| `FILL_IN_ACCOUNT` | Your Slurm account name (from Step 0) |
| `FILL_IN_GPU_PARTITION` | GPU partition name (from Step 0) |
| `FILL_IN_REPO_DIR` | Absolute path to this cloned repository |
| `FILL_IN_DATA_DIR` | Directory containing the `signal_mX*.hdf5` files from Step 2 |

Also add any `--qos` or other cluster-specific flags your cluster requires.

### Submit

```bash
mkdir -p logs
sbatch submit/submit_e022_portable.sh
sbatch submit/submit_e023_portable.sh
```

Both scripts are **self-resubmitting**: each job schedules its own continuation before running, and checks `training_state.json` at startup to exit cleanly if `done=true`. They will chain themselves automatically until 200 epochs.

### Verify the first epoch

After the first epoch completes (~10-30 min depending on GPU speed), check:
```bash
# E022 — val_loss should be ~4.9
python3 -c "import json; print(json.load(open('FILL_IN_DATA_DIR/checkpoints_bsm_grid/bsm_grid_event_c_layers4/training_state.json')))"

# E023 — val_loss should be ~5 (similar to early E020c training)
python3 -c "import json; print(json.load(open('FILL_IN_DATA_DIR/checkpoints_bsm_grid/bsm_grid_event_c_stage1/training_state.json')))"
```

If E023 val_loss is >> 10, the normalisation stats are wrong. Check that `normalisation_stats_event_c_stage1.json` is inside the checkpoint directory (not the data directory) and that `jet_std[0]` is ~0.97 (not ~1e-5).

## Experiment reference

### E022 — 4-layer generator head (bsm_grid_event_c_layers4)

- **What:** Variant of E020c with `num_gen_layers=4` (vs default 2). Tests whether a deeper particle-generator head improves fidelity.
- **Architecture:** `scripts/PET_pp_parton_vpar_bsm_event_c_layers4.py`
- **Training:** `scripts/bsm_grid_train_event_c_layers4.py`
- **Submit:** `submit/submit_e022_portable.sh`
- **Checkpoint dir:** `{CKPT_DIR}/bsm_grid_event_c_layers4/`
- **Target:** 200 epochs, patience=30, expected val_loss ~4.9 at epoch 10

### E023 — 8-dim stage-1 diffusion (bsm_grid_event_c_stage1)

- **What:** Expands stage-1 ResNet from predicting 1-dim log_npart to predicting 8-dim [log_npart, log1p(MET), sin(MET_phi), cos(MET_phi), log1p(cone_pT_X), log1p(cone_mass_X), log1p(cone_pT_Y), log1p(cone_mass_Y)]. Stage-2 particle generator is unchanged. ResNet width 512.
- **Architecture:** `scripts/PET_pp_parton_vpar_bsm_event_c_stage1.py`
- **Training:** `scripts/bsm_grid_train_event_c_stage1.py`
- **Submit:** `submit/submit_e023_portable.sh`
- **Checkpoint dir:** `{CKPT_DIR}/bsm_grid_event_c_stage1/`
- **Stats file:** `{CKPT_DIR}/bsm_grid_event_c_stage1/normalisation_stats_event_c_stage1.json` (auto-computed on first run; must be in ckpt_dir, not data dir)
- **Target:** 200 epochs, patience=30, expected val_loss ~5.0 at epoch 1

## Coding conventions

- TensorFlow 2.15 / Keras + Horovod (optional via stub). Do not propose PyTorch.
- Minimal targeted edits over rewrites.
- No try/except to suppress errors — the existing Horovod try/except is intentional and structural.
- Do not modify `PET.py` or `layers.py` (upstream OmniLearn framework).
- Preserve argparse structure in all training scripts.
- Stats files must live in `ckpt_dir`, never in `grid_dir` (contamination risk if smoke-testing against production data).

---

## Known pitfalls

- **Horovod LR scaling:** The training scripts scale `lr *= hvd.size()` to keep the effective learning rate constant across workers. With the single-GPU stub (`hvd.size()=1`), no scaling happens — this is correct for single-GPU.
- **Stats contamination:** Never run a smoke test with `--grid_dir` pointing at the real data directory while using a temp `--ckpt_dir`. The E023 stats default to `ckpt_dir` specifically to prevent this.
- **E023 normalisation_stats_event_c_stage1.json must have `jet_std[0] ≈ 0.97`** (log_npart standard deviation). If it shows ~1e-5, it was computed on a tiny dataset and must be deleted and recomputed.
- **File naming:** The data loader globs for `signal_mX*.hdf5` in `grid_dir`. Generator output follows `signal_mX{mX:04d}_mY{mY:04d}.hdf5`. Do not rename files.
